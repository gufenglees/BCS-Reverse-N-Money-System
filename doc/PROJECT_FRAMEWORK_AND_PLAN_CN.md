## 1. 项目总览

BCS 逆向货币离线支付系统是一个以 Bidirectional Currency System 为理论基础的轻量级区块链工程原型。MVP 阶段把论文中关于需求侧价值与 N 货币双向约束流转的设想，转换为“外部金额 + 链上 N 结算”的可运行系统；现实支付凭证只是可选审计信息。它的核心不是复制传统公链的开放挖矿模型，也不是前期发行链上 D 资产，而是面向授权节点、低算力场景、离线优先支付、身份认证和 N 规则强约束的专用链系统。

从工程形态看，当前项目以 Python 为主语言，围绕 `bcs_chain/` 包组织代码，包含核心链、交易、UTXO、共识、货币规则、身份、离线同步、ZK、API、P2P、钱包、CLI、Docker、仿真与测试等模块。它已经具备一个完整系统的骨架: 有数据模型、有模块接口、有 REST/gRPC 服务、有钱包能力、有测试目录、有容器化配置，也有面向后续产品化的架构设计文档。

从业务形态看，MVP 阶段系统只把 N 货币作为链上资产处理。D 侧不作为链上货币发行或托管，而是被抽象为 `external_amount` 这个销售/工资规则计算基数。现实货币、银行转账、现金、发票、工资单、支付网关或稳定币网关等只是可选外部支付引用，用于审计、对账或后续验证接口，不是当前共识层强制字段。销售和工资规则仍然保留 `phi`、`psi`，但它们只根据外部支付金额计算需要转移的 N。这样既保留 BCS 经济约束，又避免前期同时处理两种链上资产。

从技术目标看，项目有几个鲜明特点。第一，采用 UTXO 模型而不是账户模型，主要是为了支持离线交易、自包含验证、并行处理和双花检测。第二，采用 PoA 或 PoA-BFT 共识，不做高能耗挖矿，适合许可网络。第三，内置 DID/VC 身份层，控制用户认证、N 初始发放、后续补充、治理权限等关键动作。第四，支持离线支付缓存、乐观 UTXO 视图、重连同步和冲突解决，这是项目区别于普通链原型的重点。第五，保留零知识证明模块，为未来隐私转账和比例证明提供扩展方向。

当前项目最适合定位为“完整架构原型”或“MVP 前后的工程样机”。它已经覆盖大部分系统边界，但其中仍有不少模块处于原型、简化、桩实现或教学级实现阶段。要进入真实生产，还需要重点补齐依赖锁定、导入体系收敛、数据序列化安全、治理权限、密钥管理、生产配置、TLS/mTLS、协议版本化、可验证状态证明、离线冲突闭环、端到端测试和性能压测。

## 2. 业务背景与系统目标

### 2.1 业务背景

传统货币系统通常以支付、清算、价值尺度和储藏功能为中心。区块链系统又进一步把记账权、交易验证、资产流转和不可篡改账本放到分布式系统中。然而 BCS 所代表的逆向货币模型并不是简单地发行一种新 Token，而是希望把“需求”与“被需要”这两个方向同时放入交易结构。MVP 中需求侧价值由 `external_amount` 表达，现实支付或外部凭证可选附带，N 货币表达经济主体在系统中的被需要约束。

在销售交易里，消费者支付 D，商家获得 D，但商家需要向消费者转移一定比例的 N。该设计意味着销售行为不是单向获得价值，而是伴随一部分 N 的释放。商家的销售规模会被其 N 持有量约束。如果商家缺少 N，则不能无限扩大销售，除非通过其他规则获得补充。这样，N 成为一种调节销售能力、反馈经济关系、限制失衡扩张的工具。

在工资交易里，雇主支付 D，工人获得 D，但工人需要向雇主转移一定比例的 N。该设计使劳动关系也进入双向约束。工资不是单向资金流，而伴随 N 的反向流动。通过 `psi` 参数，系统可以表达劳动场景中不同于销售场景的 N 流转比例。

这种模式对软件系统提出了不同于普通支付链的要求。系统不能只检查余额和签名，还必须理解交易类型、参与方角色、外部支付金额、N 金额、比例参数、身份状态、治理权限和离线同步状态。换句话说，业务规则不是应用层的附加说明，而是共识验证的一部分。

### 2.2 总体目标

项目总体目标是实现一个轻量级、离线优先、身份约束、规则可治理、可仿真评估的 BCS 专用链系统。它不追求成为通用公链，不追求开放挖矿，也不追求高复杂度智能合约平台，而是围绕 BCS 经济模型提供一条可验证、可部署、可集成的业务链。

具体目标可以拆为八类:

第一，提供链上 N 货币的发行、转移、销毁、补充和生命周期管理。N 货币必须可追踪、可约束、可审计，并能与身份状态和治理规则绑定。

第二，提供外部支付金额相关的交易规则验证。MVP 不发行链上 D 资产，但 `external_amount` 会作为交易 metadata 参与验证，保证 `phi` 与 `psi` 的 N 义务能被共识层执行。银行、现金、支付网关、发票、工资单等引用可以附带，但不是必填项。

第三，提供离线支付能力。用户或节点在短时间无法联网时，仍可以根据本地 UTXO 快照创建交易、签名、缓存，并在重连后同步。系统需要检测双花、规则变化、余额变化、过期等冲突。

第四，提供轻量共识。系统面向授权节点网络，使用 PoA 或 PoA-BFT，以固定验证者集合或治理可变验证者集合出块，不需要 PoW 挖矿。

第五，提供身份认证。用户通过 DID 和 VC 完成身份绑定，认证状态决定是否可获得 N 初始发放、是否可参与特定交易、是否可被治理约束。

第六，提供隐私扩展能力。普通模式下交易公开；隐私模式下使用 commitment、nullifier、range proof、ratio proof 等零知识组件隐藏敏感金额，同时保留规则可验证性。

第七，提供 API、CLI、钱包和容器化部署。系统既要能被开发者调用，也要能被运维部署，还要能被仿真程序驱动。

第八，提供测试和大规模仿真能力。BCS 经济模型需要观察动态行为，因此项目不只要有单元测试，还要有模拟交易、压力测试、离线恢复、网络分区和经济指标统计。

### 2.3 非目标

为了控制系统复杂度，本项目当前阶段不应把以下内容作为优先目标。第一，不做通用虚拟机或完整智能合约平台。第二，不做匿名公链式开放验证者加入。第三，不把 ZK 模块直接宣称为生产级隐私系统。第四，不在 MVP 阶段追求全球规模吞吐。第五，不把所有业务规则外包给链下服务，因为那会削弱共识约束。第六，不在没有密钥管理和合规审计的情况下直接生产部署。

这种非目标划定很重要。项目的竞争力在于“离线优先 + BCS 规则 + 身份治理 + 轻量链”，而不是与通用链在生态、合约、吞吐或金融组合能力上竞争。

## 3. 总体架构框架

### 3.1 架构分层

系统可以分成九个主要层次:

第一层是客户端层，包括 CLI、钱包、SDK、REST 客户端和可能的移动端或嵌入式设备。该层负责密钥管理、地址生成、余额查看、交易构建、离线模式管理、导入导出等用户侧能力。

第二层是 API 层，包括 REST API 和 gRPC API。REST 适合 Web、管理后台、钱包集成和调试；gRPC 适合节点间同步、流式区块下载、轻客户端证明获取和高性能调用。API 层需要做输入校验、错误转换、状态查询、提交交易和节点健康检查。

第三层是节点运行时层，即 `BCSNode`。它把 core、currency、identity、offline、network、api、wallet、zk 等模块组合成一个完整节点进程。节点运行时负责加载配置、初始化存储、创建共识对象、启动 REST/gRPC/P2P 服务、处理出块循环和优雅停止。

第四层是区块链核心层，包括交易、区块、UTXO、状态、验证器、mempool、共识和存储。该层是系统可信状态的基础，所有链上交易最终都要通过该层验证和落盘。

第五层是 BCS 货币规则层。它不只是普通 Token 逻辑，而是实现 `phi`、`psi`、N 可行性、N 生命周期、治理参数等业务规则。该层与交易验证层紧密耦合，决定特定交易是否符合经济约束。

第六层是身份与权限层，包括 DID 文档、VC 凭证、身份注册表、信任锚和认证引擎。该层控制用户是否已认证、是否可接收 N、治理者是否有权限、DID 是否有效。

第七层是离线支付层，包括离线交易构建、本地缓存、乐观 UTXO 视图、轻客户端证明、同步引擎和冲突解决器。该层使系统能在网络不稳定或短时断网场景下继续运行。

第八层是网络层，包括 P2P 节点、peer 管理、消息结构、gossip、区块和交易广播。该层负责节点间传播和同步。

第九层是隐私与证明层，包括 commitment、nullifier、prover、verifier 和电路定义。该层目前适合作为原型扩展，为后续生产级 ZK 集成预留接口。

### 3.2 架构关系

一个典型在线交易的路径是: 钱包构建交易，交易通过 CLI 或 REST API 提交到节点，节点 API 将请求转换为核心交易对象，交易验证器检查签名、输入、输出、UTXO、脚本和 BCS 规则，mempool 接收交易，P2P 网络广播交易，PoA 共识选择交易打包成区块，区块验证通过后写入存储，并更新 UTXO 集、账户派生状态、身份状态和索引。

一个典型离线交易的路径是: 钱包基于最近一次同步的 UTXO 快照选择输入，离线构建交易并签名，交易进入本地 SQLite 缓存，同时乐观更新本地 UTXO 视图。重连后，SyncEngine 找到链上最新状态，检测本地交易引用的输入是否已被花费，检测规则参数是否变化，检测 TTL 是否过期，能提交的交易批量提交，冲突交易交给 ConflictResolver 尝试重建、调整或拒绝。

身份相关交易路径是: 用户生成密钥和 DID，信任锚签发 VC，用户提交身份注册交易，节点验证 DID 文档、VC 签名、过期时间和发行者权限，身份注册表更新状态。认证完成后，用户可以接收 MINT 或参与特定权限交易。

治理相关路径是: 治理节点或治理多签发起参数变更、验证者集变更、N 货币发放或销毁交易，验证器检查多签阈值、提案状态、生效高度和参数范围，确认后写入链上，后续交易验证读取新的系统参数。

### 3.3 架构优势

该架构最明显的优势是职责分离较清楚。核心链不直接承担身份、ZK、钱包和 API 的所有细节；货币规则作为独立模块存在；离线能力单独成层；仿真与生产运行也分离。这种结构有利于逐步替换原型模块。例如未来可以把 ZK prover 替换为 Rust FFI，把存储替换为 RocksDB，把 P2P 替换为 libp2p，而不必重写所有业务层。

第二个优势是 UTXO 模型与离线支付目标匹配。离线交易最怕全局 nonce 顺序、账户余额竞争和重连后难以解释冲突。UTXO 输入引用天然提供双花检测边界，同一 UTXO 是否已花费非常明确。对于离线设备来说，只要持有 UTXO 证明和最近状态根，就能构建较自包含的交易。

第三个优势是规则可验证。BCS 的经济规则不是钱包端提示，而是交易验证的一部分。只要所有节点执行同一规则，错误比例或违规 N 流向的交易就不能进入有效区块。

第四个优势是扩展空间充足。身份、ZK、治理、网络、仿真等模块都已具备接口和基础实现，后续可以逐步从原型向生产推进。

### 3.4 架构风险

该架构也存在明显风险。第一，当前代码里仍有不少历史脚本式导入、桩实现和 `_core_stubs`。这说明工程处在从实验脚本向正式包过渡的阶段，短期内应优先清理导入和数据模型漂移。

第二，ZK 模块和 light client 证明还不能直接视为生产级安全实现。它们更像接口原型和教学实现，后续需要严格密码学审计。

第三，默认配置偏开发环境，例如 `0.0.0.0` 和通配 CORS。生产部署必须增加配置校验和 secrets 管理。

第四，离线交易虽然有缓存和冲突解决框架，但要达到真实支付体验，还需要端到端测试、多设备场景、用户提示、冲突补偿和同步失败恢复策略。

第五，身份和治理是系统安全核心，如果治理多签、信任锚吊销、身份状态迁移和参数生效高度没有严格实现，攻击面会很大。

## 4. 工程目录与代码分层

### 4.1 根目录文件

根目录包含 `architecture_design.md`、`plan.md`、`PROJECT_FRAMEWORK_AND_PLAN_CN.md`、`pyproject.toml`、`.gitignore`、`.dockerignore`、`LICENSE` 和 `bcs_chain/`。其中 `architecture_design.md` 是原始完整架构设计文档，包含系统概述、数据模型、API、部署、安全和代码骨架。`plan.md` 是阶段化实施计划，说明从架构设计、核心链、离线支付、ZK、身份、客户端、测试、部署到仿真的阶段安排。本文件则是对项目框架、详细方案和后续评价的中文汇总。

`pyproject.toml` 是新增的标准 Python 项目描述文件，声明项目名、版本、依赖、开发依赖、CLI 入口、pytest 配置和 ruff 基础规则。它的价值是让项目不再只依赖 `requirements.txt`，而可以用 `pip install -e ".[dev]"` 进入开发状态，并为后续打包、CI 和工具链统一提供基础。

`.gitignore` 与 `.dockerignore` 分别用于版本库和 Docker 构建上下文清理。当前项目中曾存在大量 `__pycache__`，这类运行产物不应进入版本管理，也不应进入镜像上下文。忽略规则覆盖 Python 缓存、虚拟环境、构建产物、数据库、钱包文件、validator keys、genesis 文件、日志、本地环境变量和编辑器目录。

`LICENSE` 补齐 MIT 许可证文件，使 README 的许可证声明有实际承接。

### 4.2 `bcs_chain/core`

`core` 是系统最基础的链模块。它包含:

- `transaction.py`: 交易结构、输入、输出、交易类型、ZK 附件和序列化辅助。
- `block.py`: 区块头、区块体、区块结构和 Merkle root 计算。
- `utxo.py`: UTXO 数据结构、UTXOSet、PatriciaTrie 或 Merkle root 相关能力。
- `state.py`: 账户派生状态、身份状态和 StateManager。
- `script.py`: 简化脚本引擎，支持 P2PKH、多签、治理签名和 DID 检查等操作码。
- `validator.py`: 交易验证、区块验证、系统参数和验证结果。
- `mempool.py`: 内存交易池、交易排序和入池管理。
- `consensus.py`: PoA-BFT 共识、验证者集合和出块逻辑。
- `storage.py`: 区块存储、索引存储和本地数据库接口。

该层应该保持最小可信核心原则。凡是会影响账本有效性的规则，都必须在 core 或 core 调用的规则模块中确定性执行。任何依赖当前时间、外部 API、随机数或本地配置的逻辑，都不能直接进入共识验证路径，除非被转化为区块字段或治理参数。

### 4.3 `bcs_chain/currency`

`currency` 负责 BCS 经济规则:

- `params.py`: 系统参数、治理参数、比例配置和参数变更。
- `rules_engine.py`: 货币规则引擎，验证销售交易、工资交易、N 转移、规则比例和输出流向。
- `feasibility.py`: N 可行性检查，计算可销售容量、已用容量和补充建议。
- `n_lifecycle.py`: N 的铸造、补充、销毁、冻结、撤销等生命周期管理。

这个模块是项目业务价值的核心。普通 UTXO 链只关心输入是否存在、签名是否正确、输出是否不超过输入；BCS 还要关心销售场景中卖家是否按 `phi` 给买家 N，工资场景中工人是否按 `psi` 给雇主 N，N 发放是否满足身份和治理要求，N 持有量是否支持销售规模。

### 4.4 `bcs_chain/offline`

`offline` 是离线优先能力的集中实现:

- `tx_builder.py`: 离线交易构建和签名。
- `cache.py`: SQLite 离线交易缓存，支持 TTL、状态机、sequence。
- `utxo_view.py`: 本地乐观 UTXO 视图。
- `sync.py`: 重连同步引擎。
- `conflict_resolver.py`: 冲突类型识别和自动/半自动解决。
- `light_client.py`: 轻客户端证明和头链验证。
- `_core_stubs.py`: 当前用于离线模块的最小核心数据桩。

该模块目前是项目最具特色也最需要持续打磨的部分。离线支付不是简单把交易存到本地再发出去，而是要处理断网期间链上状态变化、双花、参数变更、输入过期、身份状态变化、交易优先级和用户体验。后续应把 `_core_stubs` 逐步替换为真实 core 模型或清晰的 DTO，以减少模型漂移。

### 4.5 `bcs_chain/identity`

`identity` 包含:

- `did.py`: DID 文档、DID 管理、密钥绑定和挑战签名。
- `vc.py`: 可验证凭证、凭证主体、证明结构和 VC 管理。

BCS 系统中的身份不是附加功能，而是 N 货币初始发放、补充、治理和权限控制的基础。身份模块需要特别关注信任锚的生命周期、凭证撤销、DID 更新、密钥轮换、身份状态与链上交易的绑定。

### 4.6 `bcs_chain/zk`

`zk` 包含:

- `commitment.py`: Pedersen commitment、nullifier 生成和基础承诺工具。

当前 ZK 模块更适合被理解为隐私模式的工程接口和原型实现。生产级 ZK 需要可信参数、可审计电路、形式化约束、第三方审计、边界测试和性能评估。项目后续应明确 public 模式和 shielded 模式的安全边界，避免原型证明被误用为生产证明。

## 5. 核心区块链层方案

### 5.1 数据结构原则

核心链的数据结构必须满足确定性、可序列化、可验证和可演进。确定性意味着同一笔交易在不同节点上产生相同哈希、相同验证结果和相同状态变更。可序列化意味着交易、区块、状态证明可以稳定传输和存储。可验证意味着轻客户端和其他节点可以用证明确认交易包含、UTXO 存在、身份状态和参数状态。可演进意味着协议字段需要版本号和扩展字段，以支持后续升级。

交易结构包括版本、交易类型、输入、输出、锁定时间、额外字段、见证和可选 ZK proof。输入引用前序交易哈希和输出索引，输出包含金额、锁定脚本、资产类型和元数据。区块结构包括区块头和区块体，区块头包含前序区块哈希、交易 Merkle root、UTXO root、身份 root、时间戳、高度、交易数量、验证者公钥、签名和额外数据。

在 BCS 场景中，交易类型非常重要。普通转账、销售、工资、铸造、补充、销毁、身份注册、身份更新、治理参数变更和验证者集合变更，都应有明确类型。验证器根据类型执行不同规则。

### 5.2 UTXO 模型

系统选择 UTXO 模型有充分理由。首先，UTXO 适合离线交易。用户离线时只需要证明自己拥有某些未花费输出，并构造引用这些输出的交易。账户模型则需要依赖 nonce 和账户余额的全局顺序，离线并发更容易冲突。

其次，UTXO 天然支持双花检测。一个输出只能被花费一次，节点只需检查输入引用是否仍在 UTXO 集中。重连后，如果某个离线交易引用的 UTXO 已被链上交易花费，则冲突原因明确。

第三，UTXO 支持多输入多输出，适合 BCS 的多角色流向。例如销售交易可能需要卖家输入、买家输出、找零输出、N 回馈输出、手续费输出。工资交易也类似。

第四，UTXO 与隐私模型更匹配。commitment/nullifier 体系通常围绕 note 或 UTXO 建模，隐藏金额和所有者时通过 nullifier 防双花。

UTXO 的缺点是状态集合可能膨胀，查询账户余额需要索引，复杂交易构造难度高。项目通过 UTXOSet、地址索引、派生账户状态和存储层来缓解。

### 5.3 交易验证流程

交易验证可分为基础验证、脚本验证、余额验证、规则验证、权限验证和隐私验证。

基础验证检查版本、类型、字段长度、输入输出数量、金额非负、锁定时间、序列化格式和哈希格式。脚本验证检查每个输入的解锁脚本是否满足被引用输出的锁定脚本，例如签名、公钥哈希、多签或 DID 绑定。余额验证检查输入金额总和是否大于等于输出金额总和加手续费。规则验证根据交易类型检查 `phi`、`psi`、外部支付金额、N 输出流向和可行性。权限验证检查 MINT、REPLENISH、BURN、GOV_PARAMETER_CHANGE 等交易是否有治理签名或身份权限。隐私验证检查 ZK proof、nullifier 未使用、commitment 有效和公开输入符合规则。

所有验证都应返回结构化结果，而不是只返回布尔值。结构化结果应包含错误码、错误原因、可重试标记、冲突字段和用户可读提示。这样钱包和离线同步器才能做自动处理。

### 5.4 区块验证流程

区块验证包括区块头验证、前序链接验证、高度验证、时间戳验证、验证者签名验证、交易 Merkle root 验证、交易列表验证、状态 root 验证和共识规则验证。

PoA 场景下，验证者签名是关键。节点必须确认出块者在当前高度属于有效验证者集合，签名覆盖正确的 header 字段，区块高度符合轮转或共识规则。如果验证者集合变更交易在某高度生效，则区块验证要读取对应高度的 validator set。

交易列表验证不仅要逐笔验证，还要检测同一区块内双花。如果两笔交易引用同一 UTXO，即使该 UTXO 在区块前有效，也只能有一笔成功。区块应用交易时应按确定性顺序更新临时 UTXO 视图，最后得到新的 UTXO root 和状态 root。

### 5.5 Mempool

Mempool 是已验证但未上链交易的临时池。它需要处理交易入池、去重、按手续费或优先级排序、依赖交易、冲突交易、过期交易和区块确认后的移除。对离线交易而言，mempool 还应能够识别同一输入的冲突提交，并为冲突解决提供原因。

Mempool 的设计不应改变共识规则。mempool 可以更严格，例如拒绝低费、过期、重复或暂时不可打包的交易；但区块验证必须独立于本地 mempool 策略。否则不同节点 mempool 策略不同可能导致共识不一致。

### 5.6 共识方案

项目采用 PoA-BFT 思路，适合授权验证者网络。PoA 的核心是已知验证者轮流或按规则出块，区块由验证者签名确认。BFT finality 则要求一定比例验证者签名或确认，典型是超过三分之二。

PoA 的优点是低能耗、出块快、实现简单、适合联盟链或许可链。缺点是去中心化程度较低，对验证者身份、治理和密钥安全依赖强。BCS 的业务模型本身包含身份和治理，因此 PoA 是合理选择。

共识模块后续应重点补齐四类规则。第一，验证者集变更的生效高度，避免不同节点在同一高度使用不同集合。第二，视图切换或出块者失效处理，避免某验证者离线导致停链。第三，最终性证明结构，方便轻客户端确认区块不可回滚。第四，惩罚或信誉机制，处理双签、长时间不出块、恶意区块等行为。

### 5.7 脚本系统

项目设计了简化版脚本系统，类似 BTC 但更可审计。标准脚本包括 P2PKH、多签、治理签名和 DID 绑定。脚本系统的原则是足够表达支付条件，但不要扩展成通用虚拟机。这样可以降低验证复杂度和攻击面。

P2PKH 用于普通地址支付，多签用于治理和机构账户，DID 绑定用于身份相关输出。`OP_CHECKGOVSIG` 和 `OP_CHECKDID` 是 BCS 专用能力，但这些操作码不应直接读取本地非共识数据。它们需要依赖链上状态或区块高度对应的状态 root。

## 6. BCS 货币规则与经济模型方案

### 6.1 外部支付与 N 的角色

MVP 中，链上只处理 N。外部支付金额代表需求侧价值，可对应现实货币、银行转账、现金、发票、工资单、支付网关或未来稳定币/银行接口，但这些外部引用只作为可选审计线索。N 货币是链上原生资产，系统可以精确追踪它的 UTXO 流转。BCS 的关键不再是“链上 D + 链上 N”双资产，而是“外部金额 + 链上 N 义务”的绑定。

销售交易中，卖家通过外部支付系统获得现实价值，但必须向买家转移至少 `phi * external_amount` 的 N。工资交易中，工人通过外部支付系统获得工资，但必须向雇主转移至少 `psi * external_amount` 的 N。这样，N 成为经济关系中的反向反馈资产。

在工程实现中，外部支付金额必须放入交易额外字段或特定 schema 中，而币种、银行流水、现金备注、支付网关订单号、发票哈希、工资单哈希等引用是可选字段。N 金额通过输出金额体现。验证器必须能识别交易角色和对应 N 输出。角色识别可以通过交易 metadata、标准输出结构、DID 角色声明或 API schema 固定格式实现。生产系统应避免仅靠自由文本 metadata。

### 6.2 销售规则

销售规则的核心是:

```text
N_to_buyer >= phi * external_amount
```

其中 `phi` 是治理参数，通常用分子/分母表示，避免浮点误差。若 `phi = 3/100`，外部支付金额为 1000，则卖家至少需要给买家 30 单位 N。由于系统使用 nanoN 或整数最小单位，计算时应明确向上取整还是向下取整。对规则约束而言，通常应向上取整，保证不会因除法截断低于最低要求。

销售交易验证需要知道卖家、买家、外部支付金额和 N 输出。验证器应检查卖家输入中有足够 N，输出中有至少一项锁定给买家，金额满足比例，找零输出合理，手续费不导致输出超过输入。如果隐私模式启用，则需要 ZK proof 证明隐藏金额仍满足比例。

销售规则还与 N 可行性约束相关。商家的最大销售能力可由其可用 N 余额除以 `phi` 估算。若商家 N 不足，其销售规模受限。该机制是 BCS 的核心经济效果之一。

### 6.3 工资规则

工资规则的核心是:

```text
N_to_employer >= psi * external_amount
```

其中 `psi` 是工资场景比例。工资交易中，雇主通过现实支付系统发薪，工人需要向雇主转移 N。工程上需要明确外部工资支付如何证明。链上交易必须携带外部支付金额；付款引用或外部凭证哈希可选。没有凭证引用时，节点仍能验证 N 比例，但不能证明现实支付是否真实发生。

工资交易同样需要角色字段。雇主、工人和 N 接收方必须明确，否则恶意交易可以伪造输出结构绕过规则。生产系统中建议使用标准化交易 schema，并由 API 层和验证器双重校验。

### 6.4 N 货币生命周期

N 生命周期包括不存在、铸造、活跃、可转移、补充、冻结、销毁和撤销。MINT 通常用于初始发放，REPLENISH 用于后续补充，BURN 用于销毁，FREEZE/REVOKE 可用于风险控制。当前交易类型中已有 MINT、REPLENISH 和 BURN，冻结和撤销可以通过身份状态或 UTXO metadata 扩展。

N 发放不应是任意地址自由铸造。它应绑定身份认证、治理多签、发放额度、周期限制和总量上限。后续补充也应有明确规则，例如身份仍有效、余额低于阈值、距离上次补充超过一定高度、治理签名满足阈值。

N 销毁可用于纠错、制裁或经济调节，但必须严格治理。任何能减少他人 N 余额的操作都应有高阈值、多签、审计事件和延迟生效机制。

### 6.5 参数治理

`phi`、`psi`、区块间隔、最大区块大小、N 上下限、治理签名阈值、补充阈值等都是系统参数。参数不能通过普通配置文件随意改变，否则节点之间会分叉。所有影响交易有效性的参数都必须上链治理，并在明确高度生效。

参数变更流程建议为: 创建提案、收集治理签名、进入等待期、在指定高度生效、记录事件。等待期能给离线节点和客户端时间同步新参数。离线交易如果基于旧参数创建，在新参数生效后重连，可能需要冲突解决。

### 6.6 可行性约束

N 可行性约束用于评估账户或商家的可销售容量:

```text
max_sale_capacity = n_available / phi
remaining_capacity = max_sale_capacity - current_period_usage
```

其中 current_period_usage 可用滑动窗口统计。该机制防止某主体在 N 不足时继续扩大 D 销售。它也为补充机制提供依据: 若销售能力不足且身份有效，可以申请补充 N 或通过市场获得 N。

可行性约束要注意统计口径。如果只看当前 N 余额，商家可以通过短时借入 N 绕过限制。如果看滑动窗口和历史使用，则更稳定但实现更复杂。系统需要明确经济目标，再决定统计模型。

## 7. 离线支付与同步方案

### 7.1 离线支付目标

离线支付目标是在设备短时无网络、网络质量差、节点不可达或局部网络隔离时，仍允许用户完成交易创建、签名和本地记录，并在重连后尽可能自动同步。它不是保证所有离线交易最终成功，而是保证离线交易有清晰状态、可验证依据、可解释冲突和可恢复流程。

离线支付的核心挑战有五个。第一，断网期间 UTXO 可能已被其他设备或链上交易花费。第二，断网期间治理参数可能变化，导致原本满足规则的交易不再满足。第三，交易可能超过 TTL 或锁定时间。第四，本地设备可能创建多笔互相依赖或互相冲突的交易。第五，用户需要知道哪些交易成功、哪些失败、失败原因是什么、能否重建。

### 7.2 本地 UTXO 视图

离线设备需要维护最近同步的链上 UTXO 快照，并在本地交易创建后乐观更新。这个视图可以看成三层:

第一层是已知链上 UTXO，即最近一次同步时确认未花费的输出。第二层是本地已花费集合，即离线交易已经引用的输入。第三层是本地创建集合，即离线交易产生但尚未上链的输出。

有效视图等于链上 UTXO 加本地创建 UTXO，再减去本地已花费 UTXO。钱包选择输入时应该使用有效视图，避免本地重复花费同一 UTXO。

这种乐观视图只能保证本地一致，不能保证链上一致。重连后必须与真实链状态 reconcile。

### 7.3 离线交易缓存

离线交易缓存建议使用 SQLite。每条记录包含交易哈希、交易数据、状态、创建时间、过期时间和序列号。状态机可包括 Draft、SignedLocal、Cached、PendingNetwork、Confirmed、Rejected、Conflicted 和 Resolved。

缓存必须保证崩溃恢复。设备断电或应用退出后，未同步交易不能丢失。缓存还应支持批量查询、按 sequence 排序、过期清理、状态更新和冲突标记。

当前 `_core_stubs` 中的交易序列化使用 `pickle`，这是后续必须优先修复的安全风险。离线缓存中的交易数据应改为确定性 JSON、msgpack 或协议二进制格式，并且反序列化时只构造受控数据结构，不能执行任意对象加载。

### 7.4 重连同步流程

重连同步建议分为六步:

第一，获取远端最新区块头，并与本地最后同步高度对比。第二，寻找共同祖先，处理可能的链重组。第三，下载缺失区块或状态快照。第四，验证头链、区块签名和状态证明。第五，更新本地 UTXO 视图。第六，提交本地离线交易，并记录接受或拒绝结果。

如果离线时间很短，可以增量下载区块并 replay。若离线时间很长，可以下载 UTXO 快照，再 replay 最近窗口内的区块。快照必须带有状态 root 和证明，否则轻客户端无法信任。

### 7.5 冲突类型

常见冲突包括:

- DOUBLE_SPEND: 交易输入已被链上其他交易花费。
- RULE_CHANGE: `phi`、`psi` 或其他参数在离线期间变化。
- INSUFFICIENT_BALANCE: 新链状态下余额不足。
- TIMEOUT: 交易 TTL 过期。
- IDENTITY_CHANGE: 身份状态被吊销、暂停或未认证。
- DEPENDENCY_MISSING: 本地交易依赖的前置离线交易没有成功。
- GOVERNANCE_REJECTED: 治理签名或权限不再满足。

每种冲突都应有机器可读代码、用户可读原因和可恢复建议。

### 7.6 冲突解决策略

冲突解决策略包括:

REBUILD: 使用新的可用 UTXO 重建交易。适合输入已花费但用户仍有其他余额的情况。

ADJUST: 根据新参数调整 N 输出或费用。适合 `phi` 或 `psi` 变化导致原金额不足的情况。

RBF: 用更高手续费或更高优先级替换原交易。适合 mempool 冲突但链上尚未确认的情况。

REJECT: 永久拒绝交易。适合余额不足、身份无效、交易过期且不可重建的情况。

USER_INTERVENTION: 需要用户选择。适合金额调整会影响收款方、交易语义发生变化或需要重新授权的情况。

离线支付体验的关键不是让所有冲突自动成功，而是让冲突可解释、可审计、可恢复。

### 7.7 离线安全边界

离线支付无法完全消除双花风险。只要用户完全离线，接收方无法实时确认输入是否已被其他地方花费。因此离线交易应根据金额和风险分级。小额交易可以接受较低确认强度，大额交易应要求在线确认或更强证明。

可以引入离线限额、身份信誉、设备证书、最近状态证明、交易 TTL、收款方风险提示和延迟交付机制。对于商户收款场景，商户可以根据用户身份、历史成功率和交易金额决定是否接受离线交易。

## 8. 身份认证与 DID/VC 方案

### 8.1 身份层定位

身份层是 BCS 系统的治理基础。普通公链可以允许匿名地址自由转账，但 BCS 中 N 的初始发放、补充、身份状态、治理权限和风险控制都依赖可信身份。DID 提供用户控制的标识，VC 提供信任锚签发的可验证凭证，链上身份注册表提供共识可见的身份状态。

### 8.2 DID 模型

项目设计自定义 `did:bcs:<pubkey_hash>`。用户生成 secp256k1 密钥对后，可以从公钥哈希派生 DID。DID 文档包含验证方法、公钥、服务端点和控制关系。用户用私钥签名挑战，证明自己控制 DID。

DID 的优势是用户标识与中心化账户分离。用户可以控制密钥，也可以通过 DID 文档支持密钥轮换、多设备和服务发现。风险是密钥丢失和 DID 更新机制复杂。生产系统应增加恢复机制、多签控制和撤销流程。

### 8.3 VC 模型

VC 是信任锚对用户身份或资格的签名证明。例如认证机构签发 `BCSIdentityCredential`，证明某 DID 已完成 KYC 或符合系统参与条件。链上注册时，节点验证 VC 的发行者、签名、过期时间、凭证类型和主体 DID。

VC 不应把敏感个人信息明文上链。链上可以记录凭证哈希、状态、发行者 DID 和必要字段，敏感内容留在用户侧或合规存储侧。未来可用 ZK 证明“用户拥有有效凭证”而不公开凭证内容。

### 8.4 信任锚

信任锚是可以签发 VC 的机构或治理实体。信任锚列表必须可治理、可轮换、可吊销。若某信任锚私钥泄露，系统必须能在指定高度后停止接受其新凭证，并对历史凭证采取策略。

信任锚管理建议包括:

- 发行者公钥和 DID。
- 生效高度和失效高度。
- 凭证类型权限。
- 吊销状态。
- 审计日志。
- 多签批准记录。

### 8.5 身份状态

身份状态可包括 UNAUTHENTICATED、PENDING、AUTHENTICATED、SUSPENDED 和 REVOKED。状态转换应由交易驱动，而不是只存在本地数据库。身份注册交易提交后进入 PENDING 或直接 AUTHENTICATED，取决于治理策略。吊销和暂停也应是链上交易。

身份状态影响交易有效性。例如未认证用户不能接收 MINT，已吊销用户不能申请 REPLENISH，暂停用户可能不能参与高风险交易。验证器必须在交易验证时读取对应高度的身份状态。

### 8.6 隐私与合规平衡

身份系统天然涉及隐私。BCS 需要在合规身份与用户隐私之间平衡。建议原则是: 链上只放最小必要状态，敏感信息不上链；凭证明文不公开；使用哈希承诺和可验证证明；支持凭证撤销但不公开撤销原因；对审计方开放受控查询而非全网公开个人信息。

## 9. 零知识证明与隐私方案

### 9.1 隐私需求

BCS 交易包含外部支付金额、N 金额、参与方角色和身份状态。如果全部公开，可能泄露用户收入、消费、商业规模和关系网络。因此系统预留 ZK 模块，用于隐藏金额或身份细节，同时证明交易满足规则。

隐私需求可分为三类。第一，金额隐私: 不公开具体 N 金额，但证明输入等于输出加费用。第二，比例隐私: 不公开 D 和 N 的具体值，但证明 N 满足 `phi * D` 或 `psi * D`。第三，身份隐私: 不公开完整 VC，但证明用户拥有有效身份凭证。

### 9.2 Commitment 与 Nullifier

隐私 UTXO 通常表示为 commitment:

```text
C = v * G + r * H
```

其中 v 是金额，r 是随机盲因子。链上公开 commitment，不公开 v 和 r。花费时使用 nullifier 防止双花:

```text
nullifier = PRF(sk, utxo_id)
```

节点检查 nullifier 是否已出现，若出现则拒绝。这样即使不知道被花费的具体 commitment，也能防双花。

### 9.3 N 转移证明

N 转移电路需要证明:

- 输入 commitment 由 prover 控制。
- 输入金额总和等于输出金额总和加费用。
- 每个金额在合法范围内。
- nullifier 正确生成且未被使用。
- 输出 commitment 格式正确。

这类证明是 shielded transaction 的基础。当前项目已有 commitment、prover、verifier 和 nullifier 相关模块，但生产化需要更严格的电路实现和审计。

### 9.4 比例证明

比例证明是 BCS 特有重点。销售交易需要证明:

```text
N_amount >= phi * external_amount
```

工资交易需要证明:

```text
N_amount >= psi * external_amount
```

如果 external_amount 和 N 都隐藏，则电路要证明两个私有值满足公开比例参数。若交易选择携带链外支付证明，电路还可以绑定外部凭证哈希。比例参数必须作为公开输入或由链上参数 root 证明绑定，避免 prover 使用错误参数。

### 9.5 身份绑定证明

身份绑定电路可以证明用户拥有某 DID 或 VC，而不公开完整凭证。它需要验证 VC issuer 签名、凭证未过期、subject DID 与交易签名者一致、凭证类型符合要求。生产上这类电路复杂度较高，可以先用链上公开身份状态实现 MVP，再逐步引入隐私凭证。

### 9.6 ZK 生产化要求

ZK 模块进入生产前至少需要:

- 明确采用 Groth16、PLONK、STARK 或其他证明系统。
- 固定曲线、哈希、域分隔、参数生成流程。
- 电路约束审计。
- trusted setup 或透明设置流程。
- proof version 和 circuit id 管理。
- nullifier 域隔离。
- 性能基准。
- 失败回退策略。

在这些完成前，应把当前 ZK 标为实验或原型，不应承诺生产隐私安全。

## 10. 钱包、客户端与 CLI 方案

### 10.1 钱包职责

钱包是用户与系统交互的主要入口。它负责密钥生成、加密存储、地址派生、签名、交易构造、余额查询、离线模式、导入导出和用户提示。钱包的安全性直接决定用户资产安全。

当前钱包使用密码派生密钥并加密私钥，这是合理基础。后续应继续增强密码策略、KDF 参数、密钥轮换、备份恢复、硬件密钥支持和命令行输入安全。

### 10.2 交易构建

钱包交易构建流程包括选择 UTXO、计算输出、计算找零、估算手续费、构建输入输出、生成签名、组装 witness 或 unlock_script、序列化交易。BCS 销售/工资交易还需要填充外部支付金额、角色、规则参数版本和必要 metadata；支付凭证引用是可选审计字段。

销售交易构建器应要求用户明确卖家、买家、外部支付金额和 N 回馈金额；支付凭证引用可选填写。若 N 回馈不足，应在本地阻止构建或提示。工资交易构建器类似。

### 10.3 离线模式

钱包离线模式需要提供:

- 最近同步高度和状态 root。
- 可用 UTXO 列表。
- 本地待同步交易列表。
- 冲突状态。
- 批量导入导出。
- 二维码、文件或近场传输扩展。

用户体验上，应清楚标记“未上链”“待同步”“已确认”“冲突”“已拒绝”。离线交易不能显示成最终成功，否则会误导用户。

### 10.4 CLI

CLI 适合开发、测试和运维。它应支持:

- 钱包创建、导入、导出。
- 地址和余额查询。
- 普通转账、销售交易、工资交易。
- 离线启用、离线交易创建、离线批次提交。
- DID 注册、身份状态查询。
- 治理参数查看、提案、投票。
- 节点启动和状态检查。

当前 CLI 中部分命令仍是 stub。建议后续把命令分成三类: stable、experimental、not-implemented。not-implemented 命令应返回非零退出码，避免脚本误判成功。

### 10.5 SDK

虽然当前主要是 Python 包和 CLI，但系统后续应提供 SDK。SDK 应封装交易 schema、签名、API 调用、离线缓存、错误码和状态解析。对第三方集成来说，SDK 比直接拼 REST JSON 更安全。

## 11. API 与外部集成方案

### 11.1 REST API

REST API 适合钱包、浏览器、调试工具和后台系统。核心端点包括健康检查、提交交易、查询交易、查询区块、查询账户余额、查询 UTXO、查询 mempool、离线准备、离线批量提交、冲突检测、身份注册、身份状态、治理参数和 ZK shield。

REST API 应遵循版本化路径，例如 `/api/v1/...`。返回结构应统一包含状态、数据、错误码、错误消息和 trace id。提交交易的错误要区分格式错误、签名错误、余额不足、规则不满足、权限不足、冲突、内部错误。

### 11.2 gRPC API

gRPC 适合节点间和高性能客户端。它可以提供流式区块同步、UTXO 快照下载、轻客户端证明、交易提交和状态查询。gRPC 服务应使用 protobuf 定义，而不是长期依赖手写 stub。生产环境应启用 TLS 或 mTLS，至少在跨主机通信时不能使用明文 insecure channel。

### 11.3 Schema 转换

API schema 与 core model 之间必须有明确转换。Pydantic 模型负责输入校验，core dataclass 负责共识逻辑。转换过程中要注意 bytes/hex/base64、枚举值、金额单位、可选字段和默认值。任何不明确的转换都可能导致哈希不一致或签名失败。

建议增加 schema roundtrip 测试: core -> API -> core 后哈希不变；API JSON -> core -> API 后字段不丢失。

### 11.4 外部支付接入预留

如果 D 是链外支付，API 可以支持可选外部支付凭证引用。例如银行流水哈希、支付网关订单号、发票哈希、签名收据或 oracle 证明。MVP 共识层不强制这些引用存在；高价值交易、商户对账或生产接入时，业务层或治理参数可以要求凭证哈希，并由治理认可的 oracle 或清算系统背书。

如果 D 也作为链上资产，则需要扩展 asset_type 和交易规则，使 D 和 N 都在链上 UTXO 中表达。这样验证更强，但系统复杂度增加。

## 12. P2P 网络与消息同步方案

### 12.1 网络职责

P2P 网络负责传播交易、传播区块、同步状态、发现节点和维护连接。BCS 网络不是开放匿名 gossip 网络，而更像授权节点网络。节点身份、网络 ID、协议版本和验证者权限都应明确。

### 12.2 消息类型

消息可分为交易消息、区块消息、状态消息和治理消息。交易消息包括新交易、批量交易、交易请求。区块消息包括新区块、区块请求、区块批次。状态消息包括 UTXO 快照请求、快照响应、状态 delta。治理消息包括提案、投票和治理证书。

每条消息应包含:

- protocol_version。
- network_id。
- message_type。
- message_id 或 nonce。
- timestamp 或 expiry。
- sender_id。
- payload。
- signature。

这些字段能防止跨网络污染、重放和消息混淆。

### 12.3 Peer 管理

Peer 管理包括连接、断开、重连、心跳、延迟统计、错误计数和信誉。恶意 peer 可能发送无效交易、重复消息、超大 payload 或错误区块。节点应对 peer 做限流和惩罚。

Peer reputation 应持久化，否则节点重启后恶意 peer 信誉清零。也应有恢复机制，避免临时网络故障导致长期封禁。

### 12.4 同步策略

新节点启动时需要从 bootstrap peers 获取最新高度、下载区块或快照。落后节点可以先同步 headers，再按批次下载 blocks。离线节点重连时可以使用更轻量的同步模式，只下载缺失范围或快照证明。

网络同步必须与存储和验证解耦。不能因为 peer 声称某区块有效就直接落盘，必须本地验证 header、签名、交易和状态 root。

## 13. 节点运行时与配置方案

### 13.1 节点运行时

`BCSNode` 是组合层，负责把所有模块连接起来。节点启动过程包括加载配置、初始化数据目录、打开存储、创建 UTXOSet、加载共识状态、初始化 mempool、初始化货币参数、初始化身份注册表、初始化离线模块、初始化 P2P、启动 REST/gRPC 服务和进入出块循环。

节点停止过程要优雅关闭 API、网络连接、数据库连接和后台任务。未完成的任务要取消或等待，避免数据损坏。

### 13.2 配置文件

配置使用 TOML，包含 network、consensus、storage、governance 和 api。配置项包括监听地址、端口、bootstrap peers、network_id、区块间隔、验证者 ID、公私钥、数据目录、治理参数、CORS 和限流。

开发环境可以使用简单配置，生产环境必须使用严格配置。特别是私钥不能明文放在 TOML 中，通配 CORS 不能用于公网 API，REST/gRPC 不能无 TLS 暴露到不可信网络。

### 13.3 配置安全校验

建议节点启动时增加配置安全校验:

- 生产模式禁止示例私钥。
- 生产模式禁止 `cors_origins = ["*"]`。
- 生产模式要求 TLS 或反向代理 TLS。
- 生产模式要求 data_dir 不在临时目录。
- 验证者节点必须有合法私钥。
- observer 节点不能配置 validator_privkey。
- network_id 必须与 genesis 匹配。
- 参数分母不能为 0。

这些校验能阻止明显错误配置进入运行。

### 13.4 环境变量

Docker 中通过 `BCS_CONFIG`、`BCS_DATA_DIR`、`PYTHONUNBUFFERED` 等环境变量控制运行。后续 secrets 应使用 Docker secrets、Kubernetes secrets、Vault 或主机密钥文件，而不是普通环境变量，因为环境变量可能被进程列表、日志或 crash dump 泄露。

## 14. 存储、索引与状态管理方案

### 14.1 存储对象

节点需要存储:

- 区块。
- 交易。
- UTXO 集。
- 地址索引。
- 身份状态。
- 治理参数历史。
- 验证者集合历史。
- Mempool 可选持久化。
- 离线交易缓存。
- peer reputation。

当前项目主要使用 SQLite 或简化存储。SQLite 适合原型和轻节点，但大规模生产可能需要 RocksDB、LevelDB 或专用状态数据库。

### 14.2 区块存储

区块存储应支持按高度查询、按哈希查询、范围查询、最新区块查询和重组处理。区块不可变，但索引可能随重组调整。PoA-BFT 若提供最终性，可以降低深度重组风险。

### 14.3 UTXO 存储

UTXOSet 是验证交易的关键状态。每个 UTXO 以 `(tx_hash, output_index)` 为 key，值包含金额、锁定脚本、资产类型、metadata、高度和是否 coinbase/mint。UTXO root 用于状态证明。

UTXO 存储应支持快照、回滚和批量应用。区块验证时可以在临时视图应用交易，全部成功后原子提交。

### 14.4 派生账户状态

虽然底层是 UTXO，但用户和 API 需要账户视图。账户状态包括地址、DID、N 余额、锁定 N、可用 N、最大销售容量、当前销售量、身份状态、首次认证高度、最近补充高度、nonce 和最近活跃高度。

账户状态是派生视图，必须能从 UTXO 和身份状态重建。不要把派生状态作为唯一真实来源，否则索引损坏会影响共识。

### 14.5 Schema migration

随着项目演进，数据库 schema 会变化。生产系统必须有 migration 机制，记录 schema version，支持升级和回滚。没有 migration 的存储很难长期维护。

### 16.3 生产 compose

生产 compose 设计三个验证者、两个 observer/RPC 节点、nginx 负载均衡和 prometheus。它已体现生产拓扑，但仍需要补齐 secrets、TLS 证书、真实 nginx 配置、prometheus 配置、日志采集、备份、滚动升级和健康检查可靠性。

## 17. 安全、隐私与风控方案

### 17.1 密钥安全

密钥是系统安全基础。用户私钥、验证者私钥、治理多签私钥和信任锚私钥都需要不同级别保护。用户私钥应由钱包加密存储，验证者私钥应在节点主机或 HSM 中，治理私钥应使用多签和冷存储，信任锚私钥应支持轮换和吊销。

测试网配置中硬编码私钥必须明确标记为示例，生产启动时应禁止示例私钥。CLI 不应要求用户把私钥或助记词直接放在命令行参数中。

### 17.2 序列化安全

`pickle` 是高风险反序列化机制，不应用于不可信数据。当前 `_core_stubs.Transaction.deserialize()` 使用 pickle，是 P0 级优化点。替代方案是确定性 JSON、msgpack、protobuf 或手写二进制格式。反序列化时应验证字段类型、长度和范围。

### 17.3 API 安全

REST API 应增加认证、限流、CORS 限制、请求大小限制、错误脱敏和审计日志。公网 API 不应允许任意来源 CORS。管理接口和治理接口必须鉴权。提交交易接口虽然可以开放，但要限流防 spam。

### 17.4 网络安全

P2P 消息需要签名、network_id、protocol_version、nonce 和过期时间。节点应拒绝过大 payload、未知消息类型、错误签名、跨网络消息和重复消息。peer 管理应有信誉和封禁。

### 17.5 治理安全

治理操作影响系统规则和资产供应，是最高风险区域。MINT、REPLENISH、BURN、参数变更、验证者变更都必须多签。参数变更应有等待期和生效高度。治理事件应可审计。紧急暂停能力也需要治理约束，不能由单点控制。

### 17.6 离线欺诈风险

离线交易无法完全避免双花。系统应通过限额、TTL、身份信誉、最近证明、收款方策略和重连冲突处理降低风险。高价值交易应要求在线确认。商户应用应把离线交易显示为“待结算”，而不是“最终到账”。

### 17.7 隐私风险

公开 UTXO 链会暴露交易图谱。身份层如果处理不当，会把真实身份与地址永久绑定。建议使用最小链上身份、地址轮换、凭证哈希、ZK 身份证明和访问控制审计。

## 18. 性能、容量与可扩展方案

### 18.1 性能目标

MVP 阶段可以设定 5 秒出块、1 MB 区块、数百到数千 TPS 以内的实验目标。真实吞吐取决于签名验证、UTXO 查询、数据库写入、网络传播和共识确认。由于 PoA 无挖矿，瓶颈主要在验证和存储。

### 18.2 性能优化方向

第一，交易验证并行化。UTXO 输入不重叠的交易可以并行验证，但最终应用状态要按确定性顺序。

第二，UTXO 索引优化。地址查询、余额查询和输入存在性检查需要高效索引。

第三，批量数据库写入。区块应用应使用事务批量提交。

第四，mempool 去重和冲突索引。按输入 key 建立索引可以快速发现双花。

第五，快照和增量同步。离线节点不应总是从 genesis replay。

第六，ZK proof 验证批处理。若未来使用大量 ZK 交易，需要批量验证或专用加速。

### 18.3 容量管理

UTXO 集会增长。系统可通过合并 UTXO、最小输出金额、dust 限制、周期性快照和 pruning 管理容量。历史区块可以保留完整节点，轻节点只保留 headers 和状态证明。

身份注册表和治理历史也会增长，但增长速度通常低于交易。仍需 schema migration 和归档策略。

### 18.4 扩展边界

如果未来交易量大幅增长，可以考虑:

- 分片或多区域子链。
- 状态通道或支付通道。
- 离线批量结算。
- 只把 D/N 规则结算上链，明细链下聚合。
- 使用更高性能语言重写验证热路径。

但这些都是后续阶段，不应过早引入。

## 19. 运维、监控与治理方案

### 19.1 运维目标

运维目标是保证节点稳定出块、API 可用、数据不丢、密钥不泄露、异常可发现、故障可恢复。BCS 是经济系统，运维事故可能直接影响资产和交易，因此需要严格流程。

### 19.2 监控指标

建议监控:

- 当前区块高度。
- 最新区块时间。
- peer 数。
- mempool 大小。
- 交易入池速率。
- 区块交易数。
- 共识错误数。
- REST/gRPC 请求延迟。
- 错误率。
- 数据库大小。
- 磁盘剩余空间。
- CPU/内存。
- 离线批次成功率。
- 冲突类型分布。
- N 总供应量。
- 治理参数版本。

这些指标应进入 Prometheus 或其他监控系统，并配置告警。

### 19.3 日志

日志应包含结构化字段，例如 node_id、network_id、height、tx_hash、peer_id、error_code、request_id。敏感字段如私钥、助记词、密码、完整 VC、个人身份信息不能进日志。

### 19.4 备份恢复

区块数据可以从网络恢复，但本地身份索引、离线缓存、钱包文件和配置需要备份。验证者节点的私钥备份必须安全。恢复流程要定期演练，包括新机器恢复、数据损坏恢复和密钥轮换。

### 19.5 治理流程

治理流程应文档化，包括参数提案、投票、多签、等待期、生效、回滚和紧急处理。治理 UI 或 CLI 应显示提案影响，例如 `phi` 从 3% 改到 4% 会影响销售交易最小 N 输出和商家容量。

## 20. 当前项目成熟度评估

### 20.1 已具备能力

当前项目已经具备完整架构骨架。它不是单文件 demo，而是按模块组织的系统工程。核心链、货币规则、身份、离线、ZK、API、网络、钱包、CLI、仿真和 Docker 都有对应代码。文档也比较完整，包含架构图、数据模型、接口定义和部署方案。

项目已经体现了 BCS 的关键业务规则: N-only 链上资产、外部金额计算基数、可选外部支付引用、销售 `phi`、工资 `psi`、N 可行性、N 生命周期、身份认证、离线同步和治理参数。这说明工程方向与理论模型基本一致，同时更适合前期快速导入。

离线模块的存在是亮点。很多链项目只关注在线交易，而本项目从一开始就把离线交易、缓存、同步和冲突解决作为核心能力，这与业务目标匹配。

### 20.2 主要不足

第一，工程导入体系仍有历史包袱。许多模块使用脚本式导入，需要兼容 `sys.path`。这在原型阶段可接受，但长期应迁移为明确的包相对导入。

第二，部分模块是 stub 或简化实现。尤其是 `_core_stubs`、ZK、light client、gRPC stub、CLI stub 和部分网络逻辑，需要明确标记安全等级。

第三，安全默认值偏开发环境。通配 CORS、明文 gRPC、硬编码测试私钥、pickle 反序列化、生产 secrets 管理缺失，都是进入生产前必须处理的问题。

第四，测试闭环不完整。虽然有测试目录，但当前环境缺少 pytest，且 Docker 不可用。后续需要 CI 保证每次改动都能跑测试和 compose config。

第五，文档与实际代码有轻微漂移。例如早期设计文档使用 `bcs_core/` 等路径，而实际是 `bcs_chain/core/`。这类漂移会影响维护效率。

### 20.3 成熟度分级

按工程成熟度可粗略评估:

- 架构完整度: 较高。
- 原型可读性: 较高。
- 业务模型覆盖: 较高。
- 包管理规范: 中等，已开始补齐。
- 测试可复现: 中低，需要安装和 CI。
- 安全生产性: 低到中，仍需大量加固。
- ZK 生产性: 低，目前更适合实验。
- 离线能力完整性: 中等，有框架但需 E2E 强化。
- 部署可用性: 中等，路径问题已修但需 Docker 实测。

整体而言，项目适合继续作为 MVP 原型推进，不适合未加固直接生产上线。

第一阶段，工程稳定化。目标是能安装、能测试、能启动单节点、能跑基础交易。重点处理包导入、依赖、测试、Docker 和 README。

第二阶段，核心业务闭环。目标是身份注册、N 发放、销售交易、工资交易和治理参数能在单节点或小网络中完整跑通。重点处理规则验证和权限验证。

第三阶段，离线闭环。目标是设备断网创建交易、重连同步、冲突检测和自动重建。重点处理缓存安全、UTXO 证明和 E2E 测试。

第四阶段，多节点与部署。目标是三验证者网络稳定出块，observer 提供 API，Docker 部署可复现。重点处理 P2P、共识、监控和配置安全。

第五阶段，隐私与生产加固。目标是引入可审计 ZK、TLS/mTLS、secrets、审计日志、性能压测和威胁模型。

## 22. 总结评价

### 22.1 项目价值评价

BCS 逆向货币离线支付系统的价值在于它不是单纯实现一条链，而是围绕一个具体经济模型构建了较完整的工程框架。调整后的 MVP 把链上资产收敛为 N，把现实支付金额作为规则计算基数，把支付凭证作为可选审计信息，再用 `phi`/`psi` 规则、N 可行性、身份认证、离线支付和轻量共识约束 N 的流转。这种组合有明显研究价值和原型价值，也更容易被早期用户接受。

项目的设计方向是合理的。UTXO 模型与离线支付匹配，PoA 与许可网络匹配，DID/VC 与身份约束匹配，ZK 与未来隐私需求匹配，仿真模块与经济模型验证匹配。整体架构没有偏离业务目标。

### 22.2 工程质量评价

工程已经具备较好的模块划分，但仍处于原型向规范工程过渡阶段。目录清晰、模块齐全、文档丰富，这是优点。导入混用、stub 较多、依赖未锁、测试环境不完整、安全默认值偏宽，这是短板。

本轮已补齐项目元数据、忽略规则、Docker 包路径、compose 挂载路径、包导入兼容、README 开发命令和 LICENSE。这些改动属于基础工程治理，能显著降低后续开发和部署的阻力。

### 22.3 技术风险评价

最高风险是安全边界。`pickle` 反序列化、生产默认配置、示例私钥、明文 gRPC、通配 CORS、治理权限、ZK 原型和离线双花，都是必须认真处理的问题。BCS 涉及资产和身份，不应以普通 demo 标准上线。

第二风险是模型一致性。离线模块使用 `_core_stubs`，核心模块使用真实 `core` 数据结构。如果两者长期并存，交易哈希、序列化、金额和验证规则可能出现漂移。后续必须统一模型或明确 DTO 边界。

第三风险是业务规则解释。`phi`、`psi`、外部支付金额来源、N 可行性窗口、补充规则和身份状态都需要精确定义。经济规则如果含糊，会导致不同实现产生不同结果。

### 22.4 可行性评价

从 MVP 角度看，项目可行。用 Python 实现轻量 PoA 链、UTXO、REST、CLI、SQLite 离线缓存和基础身份系统，完全可以支撑原型验证。Docker 多节点也可以作为演示和测试环境。

从生产角度看，还需要较大工作量。特别是 ZK、治理、多节点共识最终性、离线欺诈控制、密钥管理、审计、监控、性能和合规，都不能省略。

### 22.5 最终建议

建议项目下一步不要急于扩展新功能，而是先完成“稳定 MVP 闭环”。具体来说，优先保证安装、测试、单节点启动、交易验证、身份注册、N 发放、销售/工资交易、离线缓存和重连同步都可稳定运行。之后再进入多节点、生产部署和 ZK 隐私。

如果要把项目作为研究原型，应重点加强仿真、指标和经济模型分析。如果要作为产品原型，应重点加强钱包体验、API 稳定、离线冲突处理和安全配置。如果要走生产路线，应先完成威胁模型、安全审计、密钥管理、治理流程和运维监控。

总体评价: 这是一个方向明确、架构覆盖面广、业务特色突出的专用链原型项目。当前最需要的不是重构成另一个通用链，而是沿着现有 BCS 目标，把工程基础、规则一致性、离线闭环和安全生产性逐层补齐。只要后续能按优先级推进，项目具备从架构原型演进为可演示 MVP，再进一步演进为试点系统的基础。

## 附录 A: 关键业务流程详解

### A.1 用户开户注册与首次认证流程

用户进入系统的第一步不是直接获得资产，而是创建可控制身份和钱包。流程从本地钱包生成密钥开始。钱包生成 secp256k1 私钥、公钥和地址，同时根据公钥哈希生成 `did:bcs` 标识。随后钱包生成 DID Document，记录验证方法、公钥材料、控制器和可选服务端点。用户向信任锚提交必要身份证明，信任锚完成线下或外部系统审核后签发 VC。VC 的 subject 必须绑定用户 DID，issuer 必须属于系统认可的 Trust Anchor。

用户拿到 VC 后，通过 REST、CLI 或 SDK 构造 `REGISTER_IDENTITY` 交易。交易中包含 DID、DID Document 摘要、VC 摘要或必要凭证字段、签名和可验证 proof。节点收到交易后，先验证交易格式，再验证用户签名是否能证明 DID 控制权，然后验证 VC 的发行者、签名、过期时间、凭证类型和主体 DID。若验证通过，交易进入 mempool，随后被打包上链。区块提交后，IdentityRegistry 把该 DID 状态更新为 `AUTHENTICATED` 或根据流程先更新为 `PENDING`。

这一流程的关键验收点包括: DID 能由用户私钥控制，VC issuer 在信任锚列表中，凭证未过期，注册交易不可被他人重放，身份状态能被后续 MINT 或 REPLENISH 查询到。失败场景包括 VC 签名错误、issuer 被吊销、DID 与 subject 不匹配、用户签名无效、重复注册和身份已撤销。

### A.2 初始 N 发放流程

初始 N 发放是系统货币分配的关键动作，不应由普通用户自由发起。典型流程是用户身份认证完成后，治理节点或发行模块根据规则创建 `MINT` 交易。该交易指定接收地址、接收 DID、发放数量、发放原因、治理签名和可选批次信息。验证器检查接收者身份是否有效、发放数量是否在允许范围内、总供应量是否超过上限、治理签名是否达到阈值。

MINT 交易成功后，会在 UTXO 集中创建新的 N 输出。该输出可以锁定到用户地址，也可以包含额外 metadata，例如首次发放批次、锁定期或用途标记。用户钱包同步后即可看到 N 余额增加。若 MINT 失败，失败原因应明确: 身份未认证、治理签名不足、供应上限不足、接收地址格式错误或重复发放。

生产系统中，初始发放还应有审计报表。报表至少包括发放高度、tx_hash、接收 DID、数量、治理签名者、参数版本和总供应变化。这样后续可以追踪 N 供应来源。

### A.3 销售交易完整流程

销售交易是 BCS 的核心业务之一。假设商家 S 向买家 B 销售商品，外部支付金额为 1000，`phi = 3/100`，则商家至少需要向买家转移 30 单位 N。钱包或业务系统先构造销售交易请求，必须明确 seller、buyer、external_amount、n_rebate_amount 和找零地址；商品订单号、银行流水、发票哈希或付款凭证哈希属于可选审计字段。

钱包查询商家可用 UTXO，选择足够覆盖 N 回馈和手续费的输入。如果商家 N 不足，交易构建器应在本地阻止并给出短缺数量。如果商家 N 足够，构建器创建输出: 买家 N 输出、商家找零输出、可选手续费。交易 extra 中至少记录外部支付金额和角色，可选记录币种、支付凭证引用、订单哈希和规则参数版本。商家签名后提交节点。

节点验证时按顺序执行。第一，基础交易格式正确。第二，输入 UTXO 存在且未花费。第三，商家签名满足输入锁定脚本。第四，输出总额不超过输入总额。第五，识别该交易为 `TRANSFER_SALE`。第六，读取当前 `phi`。第七，按外部支付金额计算最低 N 回馈。第八，检查流向买家的 N 输出金额不低于最低值。第九，检查外部支付金额和角色格式；订单引用或付款凭证引用如果存在则做格式校验，不存在不拒绝。第十，检查可行性窗口中商家的销售容量是否足够。全部通过后交易入池并最终上链。

销售交易的边界用例很重要。例如外部支付金额为 0 应拒绝或特殊处理；N 回馈刚好等于最低值应通过；由于整数除法导致的小数应按协议取整；多输出给买家时是否可汇总要明确；买家地址与商家地址相同是否允许要明确；订单哈希重复是否允许要看业务策略；交易在离线期间参数变化后是否仍有效要由冲突解决处理。

### A.4 工资交易完整流程

工资交易与销售交易方向相反。雇主 E 通过现实支付系统向工人 W 发薪，工人需要向雇主回馈 `psi * external_amount` 的 N。构造交易时，需要明确 employer、worker、external_amount 和 n_rebate_amount。工资单哈希、银行流水或支付凭证哈希是可选审计字段，不是 MVP 共识层必填项。

工人工资交易的风险在于工人可能没有足够 N。若 N 不足，系统可以拒绝交易、提示补充 N、允许分期、或根据业务规则由雇主垫付并形成后续义务。这是经济模型层面的设计选择。当前 MVP 更适合采用“余额不足则拒绝”的简单规则。

验证流程与销售交易类似，但角色和 N 流向不同。验证器必须检查 N 输出流向雇主，而不是流向工人。参数使用 `psi` 而不是 `phi`。如果 `psi` 更新，离线工资交易可能需要调整。

### A.5 离线小额支付流程

离线小额支付适合低风险场景。用户设备在最近同步后保存 UTXO 快照和状态 root。断网时，钱包选择本地未花费 UTXO 构建交易，并把交易状态记为 `SIGNED_LOCAL`。用户确认后交易进入缓存，状态变为 `CACHED`，本地 UTXO 视图立即把输入标记为本地已花费，防止重复使用。

收款方如果也离线，只能基于发送方提供的交易、最近状态证明、身份和风险策略决定是否接受。对于小额交易，收款方可以接受“待结算”状态。对于大额交易，应要求联网确认。

重连后，钱包向节点提交交易。如果交易成功进入 mempool，状态变为 `PENDING_NETWORK`。确认后变为 `CONFIRMED`。若失败，状态变为 `REJECTED` 或 `CONFLICTED`。用户界面应显示具体原因，而不是笼统失败。

### A.6 离线冲突恢复流程

假设用户离线期间创建交易 T1，引用 UTXO X。重连后发现 X 已被链上交易 T0 花费。SyncEngine 标记 T1 为 DOUBLE_SPEND。ConflictResolver 检查用户是否还有其他可用 UTXO。如果有，则尝试使用新 UTXO 重建交易 T1'，保持收款方和金额不变，重新计算手续费并请求用户重新签名。若用户授权，T1' 提交网络，原 T1 标记为 Resolved。若没有可用 UTXO，则 T1 被拒绝，用户需要重新充值或取消支付。

若冲突原因是 RULE_CHANGE，例如 `phi` 从 3% 变为 4%，Resolver 计算新的最低 N 输出。如果原交易有足够多余 N，则可以调整 metadata 或直接提交；如果输出不足但找零足够，则可减少找零、增加买家 N 输出并重新签名；如果余额不足，则拒绝或提示补充。

冲突恢复必须保留审计链。用户应能看到原交易、冲突原因、新交易和最终结果。不能静默改变收款金额或交易语义。

## 附录 B: 核心数据契约建议

### B.1 交易字段契约

交易字段建议保持稳定:

```text
version: 协议版本
tx_type: 交易类型
inputs: 输入列表
outputs: 输出列表
lock_time: 锁定时间或高度
extra: 类型特定扩展字段
witnesses: 签名和证明
zk_proof: 可选零知识证明
```

`extra` 不应长期是任意 bytes 黑盒。建议为每类交易定义 schema。例如销售交易 extra 必填 seller、buyer、external_amount，可选 external_currency、external_payment_ref、order_hash、params_version。工资交易 extra 必填 employer、worker、external_amount，可选 external_currency、external_payment_ref、payroll_hash、params_version。治理交易 extra 包含 proposal_id、new_params、effective_height。

### B.2 哈希契约

交易哈希必须明确包含哪些字段。建议区分:

- `txid`: 不包含 witness，用于引用和签名。
- `wtxid`: 包含 witness，用于网络去重或完整交易识别。
- `sighash`: 每个输入签名所覆盖的字段。

如果当前实现只使用一个 hash，应在文档中明确排除 witness 和 unlock_script。否则不同节点可能因为 witness 编码不同产生不同 tx_hash。

### B.3 金额契约

所有 N 金额使用整数最小单位 nanoN。外部支付金额也建议使用整数最小单位，并在交易中声明 currency_code 和 scale。比例计算使用有理数:

```text
required_n = ceil(external_amount * numerator / denominator)
```

禁止在共识路径使用浮点数。Decimal 可以用于报表，但最终验证应使用整数。

### B.4 身份状态契约

身份状态应与高度绑定。查询某高度的身份状态必须可重放。建议记录:

- did。
- controller。
- status。
- valid_from_height。
- valid_until_height。
- credential_hash。
- issuer_did。
- last_update_tx。

### B.5 治理参数契约

参数记录应包含:

- params_version。
- phi_numerator。
- phi_denominator。
- psi_numerator。
- psi_denominator。
- max_block_size。
- required_gov_signatures。
- effective_height。
- proposal_tx_hash。

交易验证时应按交易所在区块高度读取有效参数，而不是读取节点当前最新配置。

| gRPC 明文传输 | 中 | 高 | P1 | TLS/mTLS |
| 离线双花 | 高 | 中 | P1 | 限额、TTL、冲突处理 |
| ZK 原型误用 | 中 | 高 | P1 | 明确实验状态和生产开关 |
| 导入体系混乱 | 高 | 中 | P1 | 迁移显式相对导入 |
| 参数生效高度不一致 | 中 | 高 | P1 | 上链治理版本化 |
| 信任锚私钥泄露 | 低到中 | 高 | P1 | 轮换、吊销、多签 |
| 数据库 schema 演进失败 | 中 | 中 | P2 | migration |
| P2P spam | 中 | 中 | P2 | 限流、签名、reputation |
| 监控缺失 | 中 | 中 | P2 | Prometheus 和告警 |

风险矩阵的意义是帮助团队排序。并不是所有问题都要同一天解决，但 P0 级风险不能拖到试点部署后再处理。尤其是反序列化、密钥、生产配置和治理权限，属于资产系统底线。

## 附录 F: 角色视角说明

### F.1 用户视角

用户关心的是钱包是否安全、交易是否成功、离线时能否支付、失败原因是否清楚。用户不需要理解 UTXO 和 PoA，但界面必须把“待同步”和“已确认”区分清楚。对于离线交易，用户应看到风险提示。

### F.2 商家视角

商家关心销售收款和 N 可行性。商家需要知道当前 N 余额能支持多少 D 销售额，什么时候需要补充 N，哪些订单因 N 不足失败。商家后台应提供销售容量报表和 N 使用历史。

### F.3 雇主视角

雇主关心工资支付和工人 N 回馈是否完成。系统应提供工资批量交易、凭证哈希、员工身份状态和失败原因。工资场景还可能需要批处理和隐私保护。

### F.4 治理者视角

治理者关心参数、供应、验证者、信任锚和风险处置。治理工具应展示当前 `phi`、`psi`、N 总供应、待处理提案、签名状态和生效高度。治理操作必须可审计。

### F.5 运维视角

运维关心节点是否出块、网络是否连通、数据库是否增长过快、磁盘是否不足、API 是否异常、证书是否过期。运维不应直接接触私钥明文。

### F.6 开发者视角

开发者关心模块边界、测试入口、数据模型和 API schema。项目应提供清晰文档、类型注解、测试数据和本地开发脚本，避免靠阅读全部源码理解流程。

## 附录 K: 实施细化与评价准则

### K.1 代码层实施准则

后续开发应遵循“小步、可验证、可回滚”的准则。每一次改动都应明确属于哪一层: core、currency、identity、offline、api、network、wallet、zk、simulation 或 deployment。不要在一次提交里同时修改交易哈希、API schema、钱包签名和 Docker 部署，因为这会让问题定位非常困难。对共识路径的改动尤其要谨慎，必须有测试证明旧交易、边界交易和非法交易的行为符合预期。

核心数据结构应优先稳定。`Transaction`、`TxInput`、`TxOutput`、`BlockHeader`、`UTXO`、`AccountState` 和 `SystemParameters` 一旦被 API、钱包、离线缓存和网络消息同时依赖，就不能随意改字段。若必须修改，应增加版本字段和兼容转换。对外 API 可以演进，但共识哈希不能因为非协议字段变化而改变。

导入体系应逐步迁移。当前通过 `bcs_chain/__init__.py` 兼容历史脚本式导入，是为了让现有模块先能稳定工作。长期目标应是包内使用显式相对导入，例如 `from ..core.transaction import Transaction`，包外使用 `from bcs_chain.core.transaction import Transaction`。迁移时可以按模块进行，每迁移一个模块就增加导入测试，避免一次性大改。

错误处理要从“抛异常给调用方”转为“结构化错误”。例如交易验证失败时，应返回 `INVALID_SIGNATURE`、`UTXO_NOT_FOUND`、`DOUBLE_SPEND`、`RULE_RATIO_TOO_LOW`、`IDENTITY_NOT_AUTHENTICATED`、`GOV_SIGNATURES_INSUFFICIENT` 等错误码。API 层再把错误码映射为 HTTP 状态码和用户提示。这样钱包、离线同步器和 CLI 都能做自动判断。

### K.2 协议层实施准则

协议字段必须写入文档。交易类型、asset_type、脚本 opcode、治理参数、P2P 消息类型和 ZK circuit_id 都应有登记表。没有登记表时，开发者容易重复使用编号或改变语义，最终导致节点不兼容。

协议升级应遵循“声明、等待、生效”的模式。先通过治理交易声明新版本或新参数，再给出 `effective_height`，最后在指定高度执行。离线节点和轻客户端需要这个等待窗口。如果参数立即生效，离线交易失败率会明显上升，也会让用户难以理解。

所有会影响共识结果的数据都必须来自区块、交易或可验证状态。不能在交易验证中读取本地 wall clock、外部 HTTP API、未上链配置文件或随机数。节点本地配置只能决定服务是否开启、端口、日志和本地策略，不能决定交易在共识层是否有效。

### K.3 安全开发准则

项目应把安全分成三类: 密钥安全、协议安全和实现安全。密钥安全解决私钥、助记词、治理签名和信任锚私钥如何生成、存储、使用和轮换。协议安全解决交易规则、双花、重放、权限、参数变更和身份吊销。实现安全解决反序列化、输入校验、依赖漏洞、日志泄露、CORS、TLS 和限流。

每个新功能都应回答四个问题。第一，谁可以调用它。第二，它依赖什么权限。第三，失败后状态如何恢复。第四，日志中会不会泄露敏感信息。对于资产相关功能，还要回答第五个问题: 攻击者能否通过构造边界输入凭空增加 N 或绕过 `phi`/`psi`。

对离线交易尤其要保守。离线功能天然降低实时验证能力，因此不应默认支持无限额度。建议协议层提供离线交易标记、TTL、最大金额、最近状态高度和风险等级。商户端或收款端可以基于这些信息制定策略。

### K.4 文档评价准则

一份架构文档是否有效，不取决于篇幅，而取决于是否能指导实现和验收。本项目文档应至少满足五个标准。第一，开发者能根据文档找到对应代码目录。第二，测试人员能根据文档写出验收用例。第三，运维人员能根据文档部署和排障。第四，安全人员能根据文档识别关键资产和攻击面。第五，产品或业务人员能理解系统为什么需要 N、D、`phi`、`psi` 和离线冲突处理。

因此，后续维护文档时不应只增加概念解释，还要同步增加字段、命令、错误码、状态机、验收标准和风险说明。尤其当代码从原型变为生产实现时，文档中的“可选”“模拟”“stub”“简化”必须及时更新，否则容易造成误用。

### K.5 MVP 验收场景

推荐用一个完整故事验收 MVP。系统启动三名用户: 商家 Alice、消费者 Bob、工人 Carol。Alice、Bob、Carol 都创建 DID 和钱包，并由信任锚签发 VC。治理节点为三人发放初始 N。Bob 通过现实支付向 Alice 购买商品，外部支付金额为 1000，Alice 按 `phi` 向 Bob 转移 N。随后 Alice 作为雇主通过现实支付系统向 Carol 发薪，Carol 按 `psi` 向 Alice 转移 N。期间 Bob 断网创建一笔小额离线交易，重连后成功提交。另创建一笔引用已花费 UTXO 的离线交易，重连后被标记为冲突并给出原因。

这个场景覆盖身份、MINT、销售、工资、UTXO、签名、规则验证、离线缓存、冲突检测、API 查询和区块确认。如果这个故事不能稳定跑通，项目还不应进入更复杂的 ZK 或大规模网络阶段。

### K.6 生产试点验收场景

生产试点应比 MVP 更严格。至少需要三验证者、两个 observer、一个 gateway、TLS API、持久化数据卷、监控和备份。试点期间应模拟验证者重启、observer 重启、网络短暂分区、用户离线、参数变更和信任锚吊销。系统应证明在这些情况下不会丢数据、不会错误发放 N、不会接受违规交易、不会让用户看到错误最终状态。

试点还应进行账务核对。每个区块的 N 输入输出变化、MINT、BURN、REPLENISH 和手续费都应可汇总。N 总供应量必须等于历史发放减销毁加补充的结果。若账务核对无法解释余额变化，则系统不具备资产系统的基本可信性。

### K.7 团队协作建议

建议按模块设定责任边界。核心链负责人维护交易、区块、UTXO、验证和存储。经济规则负责人维护 `phi`、`psi`、N 生命周期和可行性。身份负责人维护 DID、VC、信任锚和权限。离线负责人维护缓存、同步、冲突和轻客户端。API 负责人维护 schema、REST、gRPC 和错误模型。DevOps 负责人维护 Docker、配置、安全基线和监控。

跨模块变更需要设计评审。例如修改交易结构会影响钱包、API、离线缓存、网络消息、测试和文档。修改治理参数会影响规则引擎、节点配置、离线冲突和仿真。修改身份状态会影响 MINT、REPLENISH、权限和用户体验。

### K.8 长期演进评价

长期看，BCS 项目的成败不只取决于链能不能出块，而取决于经济规则是否能被用户理解、是否能被系统稳定执行、是否能在离线和在线之间保持一致、是否能在治理变化时不造成大规模混乱。如果 N 货币规则太复杂，用户会难以接受；如果规则太弱，又无法体现逆向货币设计的价值。因此产品化时需要把规则解释、容量提示、冲突原因和补充建议做成清晰的用户体验。

技术上，项目应避免陷入“所有方向都做一点”的状态。ZK、P2P、仿真、治理、身份和离线都很重要，但阶段优先级不同。最稳妥路线是先做公开透明的规则链和离线闭环，再做隐私增强。因为如果公开模式都不能稳定执行，隐私模式会让问题更难排查。

最终长期目标可以定义为: 一个经过身份认证的用户或机构，可以在网络不稳定环境下完成受规则约束的 N 交易，并用外部金额绑定现实经济活动；支付凭证引用可在需要时附带。节点网络可以轻量达成共识；治理者可以安全调整参数；系统可以审计 N 的供应和流向；用户可以清楚理解交易状态和风险。这一目标清晰、可验证，也符合项目当前架构的优势。
