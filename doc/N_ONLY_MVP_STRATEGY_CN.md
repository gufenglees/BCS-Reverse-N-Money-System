# BCS N-only MVP 策略说明

> 文档类型: 产品策略、架构调整、后续 D 支付接入预留方案  
> 核心结论: 前期系统只处理 N 货币；D 侧不是链上货币，而是销售/工资规则使用的外部金额概念。现实货币、银行、现金、支付网关、发票、工资单等外部支付引用是可选 metadata，后续按发展选择接入验证接口或链上资产。  
> 编写日期: 2026-05-01

---

## 1. 结论

把前期系统修改为“链上只处理 N 货币”是更务实的路线。

原因很直接:

1. 接入门槛更低。用户、商户、雇主和工人不需要理解或接受第二种链上 D 资产。
2. 合规压力更小。系统不直接发行或托管现实支付货币，先只处理 N 的铸造、转移、补充和销毁。
3. 工程复杂度更低。MVP 不需要处理法币充值、清算、支付牌照、稳定币托管、兑换汇率和 D 资产账务。
4. 业务更容易解释。现实世界中的钱照常走银行、现金、微信/支付宝、Stripe、发票、工资单或其他支付系统；BCS 链只负责记录和约束 N 的流转。
5. 后续扩展空间保留。D 侧可通过外部支付引用、oracle、支付网关、稳定币或链上 D 资产逐步接入。

因此推荐定义:

```text
BCS 链上资产: 只有 N
外部金额侧: 销售/工资等现实业务金额，作为 phi/psi 的计算基数
可选外部引用: 现实货币、银行转账、现金、发票、工资单、支付网关、稳定币网关等凭证或编号
链上交易规则: 根据 external_amount 计算必须转移的 N；外部引用不强制填写
```

## 2. 新系统边界

### 2.1 链上处理什么

链上只处理:

- N 货币 UTXO。
- N 普通转账。
- N 初始发放 `MINT`。
- N 补充 `REPLENISH`。
- N 销毁 `BURN`。
- 销售场景中的 N 回馈。
- 工资场景中的 N 回馈。
- DID/VC 身份状态。
- 治理参数 `phi`、`psi`。
- 可选外部支付凭证引用的哈希或编号。

### 2.2 链上不处理什么

链上前期不处理:

- 法币余额。
- D 货币余额。
- 法币充值提现。
- 银行清算。
- 支付通道资金托管。
- 汇率。
- 稳定币铸造和赎回。
- 用户现实资金 custody。

这些都留给现实支付系统或后续接口。

## 3. 交易语义调整

### 3.1 普通转账

普通转账保持不变:

```text
用户 A -> 用户 B: 转移 N
```

链上只验证 N UTXO、签名、金额守恒和手续费。

### 3.2 销售交易

旧理解:

```text
销售 = D + N 交换
```

新理解:

```text
销售 = 外部支付已经发生或待确认 + 链上 N 回馈
```

交易中不创建 D 资产，也不结算 D。交易必须带 `external_amount` 作为比例计算基数，外部支付引用可带可不带:

```json
{
  "external_amount": 10000,
  "external_currency": "CNY",
  "external_payment_ref": "invoice/order/payment hash",
  "seller": "...",
  "buyer": "...",
  "n_amount": 300
}
```

节点验证:

```text
N_to_buyer >= ceil(phi * external_amount)
```

其中 `external_amount` 是现实业务金额的最小单位，例如分、厘、订单积分单位或接口约定单位。`external_currency` 和 `external_payment_ref` 是可选字段；没有银行流水、发票或支付网关订单号时，也可以只提交金额和参与方信息。

### 3.3 工资交易

旧理解:

```text
工资 = D 工资 + N 回馈
```

新理解:

```text
工资 = 外部工资支付凭证 + 链上 N 回馈
```

交易中不发放 D，不托管工资。交易必须带 `external_amount`，工资单或支付凭证引用是可选字段:

```json
{
  "external_amount": 500000,
  "external_currency": "CNY",
  "external_payment_ref": "payroll hash",
  "worker": "...",
  "employer": "...",
  "n_amount": 25000
}
```

节点验证:

```text
N_to_employer >= ceil(psi * external_amount)
```

## 4. 为什么这样更容易导入和被接受

### 4.1 对用户更简单

用户不需要先兑换 D，也不需要理解链上 D。用户只需要知道:

```text
现实支付照常进行，BCS 钱包负责 N 的流转。
```

这比“先把现实钱换成 D，再用 D 触发 N”更容易理解。

### 4.2 对商户更容易

商户可以继续使用现有收款系统。BCS 只需要商户系统提供:

- 订单号。
- 支付金额。
- 支付凭证哈希。
- 买家 DID 或地址。
- 需要转移的 N。

不需要改变收银、发票、税务、退款和对账系统。MVP 阶段没有这些引用也可以先运行，只是审计和争议处理能力较弱。

### 4.3 对合规更安全

如果系统直接处理 D，可能马上涉及支付、储值、清算、稳定币或法币映射合规。N-only MVP 可以先把系统定位为:

```text
身份认证 + N 资产流转 + 经济关系约束账本
```

不直接碰现实资金托管。

### 4.4 对工程更快

N-only 让 MVP 的工程目标更清晰:

- UTXO。
- N 余额。
- N 交易。
- 身份。
- 治理。
- 离线同步。
- 外部支付凭证引用。

不用同时做支付网关、银行接口、汇率和 D 资产账务。

## 5. 预留 D 接入接口

虽然前期不处理 D，但要预留接口。推荐抽象为 `ExternalPaymentReference`。

```json
{
  "provider": "manual|bank|cash|wechat|alipay|stripe|stablecoin|oracle",
  "currency": "CNY",
  "amount": 10000,
  "scale": 2,
  "reference_id": "ORDER-2026-0001",
  "proof_hash": "sha3-256(payment receipt)",
  "status": "claimed|verified|settled",
  "verified_by": "did:bcs:...",
  "verified_at_height": 12345
}
```

### 5.1 MVP 阶段

MVP 阶段协议只要求:

- `external_amount`
- buyer/employer 等交易角色
- 满足 phi/psi 的 N 输出

MVP 阶段可选填写:

- `external_currency`
- `external_payment_ref`
- `provider`
- `proof_hash`

节点不证明现实支付一定发生，只验证 N 是否按外部金额满足比例。是否要求银行流水、发票、工资单或支付网关订单号，应由业务系统、商户规则或后续治理参数决定，不是当前共识层强制条件。

### 5.2 试点阶段

试点阶段可以增加:

- 支付凭证 hash。
- 商户签名。
- Trust Anchor 或支付验证方签名。
- 凭证状态。

### 5.3 成熟阶段

成熟阶段再选择:

- 银行 API。
- 支付宝/微信/Stripe 等支付接口。
- 稳定币支付网关。
- 链上 D 资产。
- Oracle 验证。

这个选择应由业务发展、监管环境和用户接受度决定，而不是在 MVP 一开始就强行确定。

## 6. 对现有系统的影响

### 6.1 不需要改的部分

以下模块保持核心逻辑不变:

- UTXO。
- N 转账。
- MINT。
- REPLENISH。
- BURN。
- DID/VC 身份。
- Trust Anchor。
- PoA 共识。
- 离线缓存。
- P2P 广播。

### 6.2 需要改口径的部分

需要从“双货币链上处理”改成“N 链上处理 + 可选外部支付引用”:

- 文档中 D 的描述。
- 销售交易说明。
- 工资交易说明。
- ZK ratio proof 说明。
- 钱包交易 extra metadata。
- 规则引擎注释和错误信息。

### 6.3 已做的代码调整

当前已调整:

- `TxType.TRANSFER_SALE/WAGE` 注释改为外部支付引用 + N 结算。
- `CurrencyRulesEngine` 文档和错误提示改为 external payment amount。
- `CurrencyRulesEngine` 支持 JSON extra 中的:
  - `external_amount`
  - `external_currency`
  - `external_payment_ref`
  - 兼容旧字段 `d_amount`
- `Wallet TxCreator` 创建销售/工资交易时只强制写入 `external_amount` 和角色，外部支付引用字段为空时不写入。
- `SystemParameters` 文档改为按外部支付金额计算 N 义务。

## 7. 推荐产品表达

不建议对用户说:

```text
你需要使用 D 货币和 N 货币。
```

建议说:

```text
现实支付照常进行，BCS 链只记录 N 货币。
交易发生后，系统根据现实支付金额自动计算并结算对应 N。
```

面向商户:

```text
你不用更换收款系统。只需要把订单金额给 BCS 钱包即可完成 N 回馈交易；有订单号、发票、银行流水或支付网关凭证时可以附带，便于审计。
```

面向治理者:

```text
系统参数 phi/psi 决定现实经济活动需要触发多少 N 流转，但链上资产仍只有 N。
```

---

## 8. 风险与注意事项

N-only MVP 最大的问题是: 链上不能自动证明现实支付一定发生。

因此需要明确:

1. MVP 阶段，外部支付引用是可选的声明式凭证。
2. 商户/雇主/用户需要自行保留现实支付证明；链上没有引用不代表现实支付不存在。
3. 高价值交易应由支付验证方或 Trust Anchor 签名。
4. 后续可接入支付 oracle 或支付网关确认。
5. 链上不承担现实资金清算责任。

这个风险是可接受的，因为它换来了更快上线、更低合规压力和更高用户接受度。

---

## 9. 最终建议

建议正式采用:

```text
阶段 1: N-only MVP
阶段 2: 可选外部支付凭证签名/验证
阶段 3: 可选支付接口接入
阶段 4: 根据发展决定是否引入链上 D
```

也就是说，D 不应在前期成为链上资产。D 在当前系统中首先是 `external_amount` 这个规则计算基数；现实货币、银行、现金、支付网关、发票、工资单只是可选引用和审计线索。只有当用户规模、商户需求、监管路径和技术基础都成熟后，再评估是否需要强制外部凭证验证或引入链上 D。
